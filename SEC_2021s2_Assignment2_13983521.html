Overview

Your  task    is   to  design  and  implement  a GUI-based  text  editor,    making   use  of  Gradle,  plugins  and scripts,  a  domain-specific    language,   and  internationalisation.  (For  bonus  marks, you  may  also    use   JNI  to  implement  one  of the  plugins.) ̡̨̢̡̣̩̣̥

The  basic    functionality   of  a  text  editor  is simple,  and  much    of   it  is  provided  for  you in  the  form    of   a  TextArea  object  (JavaFX)  or JTextArea  object  (Swing).    These   objects  will  handle  inserting  and deleting  text,  using    the   mouse  and  arrow  keys  to move  the  caret,    scrolling,   copy-and-paste,  and  even  undo/redo,  so you  don’t  need    to   implement  these  things  yourself. ̡̨̢̡̣̩̣̥

You  will need  to  add    on   several  features: ̡̨̢̡̣̩̣̥

  • Loading  and  saving  files. You  must  have    GUI   buttons  for  these  operations.  You’re strongly  advised  to    use   FileChooser  or  JFileChooser  to  allow the  user  to    select   the  filename/path  to  load/save.  Also be  aware  of    the   I18N  Requirements  for  file  IO. ̡̨̢̡̣̩̣̥

    ̡̨̢̡̣̩̣̥

  • DSL-based key  mapping.  On    startup,   your  application  must  parse  a file  called  “keymap”,    written   in  a  domain-specific  language  that expresses  how  to    handle   certain  extra  key  combinations.  Your application  must  then    make   those  key  combinations  work  as requested,  via  the    GUI.   This  is  described  further  in DSL  Requirements. ̡̨̢̡̣̩̣̥

    ̡̨̢̡̣̩̣̥

  • Plugins/scripts/API.  Your    application   must  have  a  plugin  loading mechanism,  a  script    loading   mechanism,  and  an  API  for both  plugins  and    scripts   to  interact  with  your  application. You  will  also    need   a  GUI  mechanism  for  adding and  listing  plugins/scripts    at   runtime.  This  is  described  further under  Plugin/Script  Requirements. ̡̨̢̡̣̩̣̥

    ̡̨̢̡̣̩̣̥

Your    project   must  use  Gradle,  with  separate subprojects  for  the    core   application,  each  of  the  plugins, and  the  API    declarations.   This  is  described  further  under Build  Requirements. ̡̨̢̡̣̩̣̥

Finally,  you    may   be  eligible  for  bonus  marks if  you  implement    one   of  the  plugins  in  C/C++ using  JNI. ̡̨̢̡̣̩̣̥

Plugin/Script   Requirements

Your  application  must  have  mechanisms to  load  and    run   both  plugins  and  scripts.  Both must  have  equivalent    access   to  a  common  API.  You must  also  provide    several   plugin/script  implementations  for  specific  purposes. ̡̨̢̡̣̩̣̥

1⁠3⁠9⁠8⁠3⁠5⁠2⁠1

Loading Mechanisms

The  application  must    not   have  any  plugins/scripts  loaded  when it  starts  up.    It   must  provide  a  dialog  box with  “Add  Plugin”    and   “Add  Script”  buttons,  and  which lists  currently  loaded    plugins/scripts.   (You  are  not  required  to implement  a  plugin/script    “unloading”   feature,  though  you  can  if you  want.) ̡̨̢̡̣̩̣̥

Plugins  should    be   loaded  via  the  reflection  API, with  the  user    supplying   the  fully-qualified  class  name  of the  plugin’s  class    (or   its  entry-point  class  if  it has  several  classes).    This   assumes  that  the  plugins  are already  located  in    the   JVM’s  search  path,  but  Gradle will  take  care    of   that  when  properly  configured. ̡̨̢̡̣̩̣̥

Scripts  should be  read  in    as   ordinary  UTF-8  text  files,  and run  through  an    interpreter.   You’re  advised  to  use  FileChooser or  JFileChooser  to    allow   the  user  to  locate  the script  file. ̡̨̢̡̣̩̣̥

1⁠3⁠9⁠8⁠3⁠5⁠2⁠1

API

Plugins  and    scripts   must  have  access  to  an API,  to  let    them   communicate  with  the  core  application. The  API  should    consist   simply  of  a  set  of interface  declarations.  How    many   you  have,  what  they  are called,  and  what    methods   they  contain  are  up  to you.  Overall,  though,    the   API  must  have  the  following characteristics: ̡̨̢̡̣̩̣̥

  • Plugins/scripts  must  not    have   direct  access  to  GUI  objects (anything  that  forms    part   of  JavaFX  or  Swing).  Therefore, your  API  must    make   no  mention  of  them. ̡̨̢̡̣̩̣̥

    Note ̡̨̢̡̣̩̣̥

    This  may seem  annoying  when    you   realise  that  giving  your  plugins direct  access  to    the   GUI  would  remove  a  lot of  API  design    work.   However,  the  point  of  the assignment  is  to    do   that  API  design  work! ̡̨̢̡̣̩̣̥

    Obviously,  your core  application  (the    API   provider)  will  use  JavaFX/Swing  itself. But  the  API    declarations,   and  the  plugins/scripts  that  use them,  can  and    should   be  kept  completely  separate  from it. ̡̨̢̡̣̩̣̥

    ̡̨̢̡̣̩̣̥

  • Plugins/scripts  must  report    a   human-readable  name  (other  than  their package/class  name),  to    be   listed  in  the  GUI. ̡̨̢̡̣̩̣̥

    ̡̨̢̡̣̩̣̥

  • Plugins/scripts  must be  able  to    access   and  modify  the  text  being edited,  and  the    caret   position.  The  latter  is  an integer,  where  0    means   the  start  of  the  file, 1  means  after    the   first  character,  etc. ̡̨̢̡̣̩̣̥

    ̡̨̢̡̣̩̣̥

  • Plugins/scripts  must  be able  to  alter    the   text  selection,  to  cause  a certain  range  of    characters   within  the  text  to  be highlighted. ̡̨̢̡̣̩̣̥

    ̡̨̢̡̣̩̣̥

  • Plugins/scripts  must  be    able   to  register  callbacks  for  the following: ̡̨̢̡̣̩̣̥

    • Modification  of  the    text   by  the  user. ̡̨̢̡̣̩̣̥

      ̡̨̢̡̣̩̣̥

    • Function  key  presses (F1-F12). ̡̨̢̡̣̩̣̥

      ̡̨̢̡̣̩̣̥

    • GUI  button  presses.    Specifically,   a  plugin/script  must  be  able to  cause  an    extra   button  to  appear  in  the toolbar.  It  must    supply   the  name  of  the  button (the  text  written    on   top)  and  the  callback  to be  invoked  when    the   button  is  pressed. ̡̨̢̡̣̩̣̥

      ̡̨̢̡̣̩̣̥
    ̡̨̢̡̣̩̣̥

  • Plugins/scripts  must  be able  to  ask    for   user  input  by  means  of a  dialog  box    containing   a  text  field. ̡̨̢̡̣̩̣̥

    ̡̨̢̡̣̩̣̥

Actual  Plugins  and Scripts

While  the  idea    of   a  plugin/script  is  that  they’ll be  written  by    third-parties,   you  are  also  required  to provide  some  yourself.    Specifically: ̡̨̢̡̣̩̣̥

  • A   “date”  plugin.  It  creates  a new  button  labelled    “Date”.   When  pressed,  the  current  date and  time  (for    the   correct  locale)  are  inserted  at the  caret  position.
    ̡̨̢̡̣̩̣̥

    The    fully-qualified   class  name  should  be  “texteditor.DatePlugin”. (It  will  assist    the   marker  greatly  if  you  use this  exact  name!) ̡̨̢̡̣̩̣̥

    ̡̨̢̡̣̩̣̥

  • A    “find”   plugin.  It  creates  a  new button  labelled  “Find”.    When   pressed,  or  when  the  F3 key  is  pressed,    the   plugin:
    ̡̨̢̡̣̩̣̥

    1. Asks  the  user  for  a search  term. ̡̨̢̡̣̩̣̥

      ̡̨̢̡̣̩̣̥

    2. Looks  for    the   first  occurrence  of  that  term in  the  text    being   edited,  but  only  from  the caret  position  forwards.     ̡̨̢̡̣̩̣̥

      ̡̨̢̡̣̩̣̥

    3. If   the  search  term  is  found, the  matching  text    is   selected  (highlighted).  If  not  found, nothing  else  happens. ̡̨̢̡̣̩̣̥

      ̡̨̢̡̣̩̣̥

    The    fully-qualified   class  name  should  be  “texteditor.FindPlugin”. ̡̨̢̡̣̩̣̥

    ̡̨̢̡̣̩̣̥

  • An “emoji”  script  (not    plugin).   It  monitors  changes  to  the text,  and  when    it   sees  the  substring  ":-)"  immediately prior  to  the    caret,   it  replaces  it  with  "😊".
    ̡̨̢̡̣̩̣̥

    (To emphasise:  this  particular    script   does  not  require  the  user to  press  a    special   button,  only  to  keep  typing.) ̡̨̢̡̣̩̣̥

    ̡̨̢̡̣̩̣̥

See the  Build  Requirements    for   further  information  about  where  the plugins/scripts  are  to    be   located. ̡̨̢̡̣̩̣̥

DSL  Requirements

The  application must  read  the    UTF-8   text  file  “keymap”  (no  filename extension)  on  start    up.   This  file  conforms  to  a domain-specific  language  that    (as   an  example)  looks  like  this: ̡̨̢̡̣̩̣̥

ctrl+i       = insert "    " at start of line
ctrl+shift+i = delete "    " at start of line
ctrl+d       = insert "//" at start of line
ctrl+shift+d = delete "//" at start of line
alt+t        = insert "/*TODO*/" at caret
alt+f        = insert "/*FIXME*/" at caret
alt+s        = delete "/*TODO*/" at caret
alt+g        = delete "/*FIXME*/" at caret

Specifically, there  are  zero    or   more  lines,  with  each  line consisting  of: ̡̨̢̡̣̩̣̥

  • A  key    combination,   containing  “alt”,  “ctrl”  and/or  “shift” followed  by  a    lowercase   letter,  separated  by  “+”  symbols. ̡̨̢̡̣̩̣̥
  • An equals  sign. ̡̨̢̡̣̩̣̥
  • Either  “insert”    or   “delete”. ̡̨̢̡̣̩̣̥
  • A  string,  in  double  quotes. ̡̨̢̡̣̩̣̥
  • Either the  phrase  “at start of line”    or   “at caret”. ̡̨̢̡̣̩̣̥

Whitespace  (except  in  strings)  is insignificant,  but  newlines    are   important. ̡̨̢̡̣̩̣̥

The  application  must  parse  this information,  using  a    JavaCC-generated   parser,  and  then  use  it to  set  up    keyboard   input.  The  goal  is  fairly self  explanatory.  Given    the   above  file,  we  want  the following  to  happen    while   editing: ̡̨̢̡̣̩̣̥

  • If  we  type  ctrl+i,  four spaces  should  be    inserted   at  the  beginning  of  the current  line,  and    the   caret  should  be  moved  the same  distance  ahead.    The   beginning  of  the  current  line can  be  found    by   searching  for  the  last  “\n” character  before  the    caret   (or  the  beginning  of  the file  if  there    isn’t   one). ̡̨̢̡̣̩̣̥

    ̡̨̢̡̣̩̣̥

  • If  we  type  ctrl+shift+i,  four spaces  should  be    deleted   from  the  beginning  of  the current  line  (and    the   caret  moved  four  spaces  back), but  only  if    there   are  four  spaces  there. ̡̨̢̡̣̩̣̥

    ̡̨̢̡̣̩̣̥

  • Similarly  for ctrl+d  and  ctrl+shift+d,    except   we  insert/delete  “//”  instead. ̡̨̢̡̣̩̣̥

    ̡̨̢̡̣̩̣̥

  • If  we type  alt+t,  the    string   "/*TODO*/"  should  be  inserted  at the  carot  location,    and   alt+s  should  delete  it  (if it’s  there).  Similarly    for   alt+f  and  alt+g  and  the string  "/*FIXME*/". ̡̨̢̡̣̩̣̥

    ̡̨̢̡̣̩̣̥

But,  to    reiterate,   this  is  only  an  example! The  user  can    put   whatever  they  want  in  the “keymap”  file  (within    the   syntax  rules),  and  that  determines what  your  application    should   actually  do. ̡̨̢̡̣̩̣̥

Marking  /  Good  Practice ̡̨̢̡̣̩̣̥

The 10  DSL  marks    will   be  allocated  principally  based  on how  much  of    this   you  can  get  working.  For instance,  can  your    application   handle  all  possible  key  combinations? Can  it  handle    insertions?   Deletions?  Can  it  handle  all strings?  Can  it    handle   operations  at  the  start  of the  line?  At    the   caret? ̡̨̢̡̣̩̣̥

However,  this  is  contingent  on using  JavaCC  properly.    For   instance,  don’t  try  to  use the  standard  Java    API   (StringTokenizer,  String.split(),  Pattern,  etc.)  for things  that  JavaCC    should   be  doing.  Also,  don’t  try to  make  overly-complicated    tokens   when  you  can  make  simple tokens  and  use    BNF   to  deal  with  combinations  of them. ̡̨̢̡̣̩̣̥

Note ̡̨̢̡̣̩̣̥

There’s  no  need    to   translate  these  strings  for  I18N purposes. ̡̨̢̡̣̩̣̥

Build  Requirements

Your  application    must   use  Gradle,  with  separate  subprojects for  (a)  the    core   application,  (b)  the  API  declarations, and  (c)  each    of   the  two  plugins.  You  will need  to  set    up   dependencies  to  ensure  everything  compiles and  runs  as    expected. ̡̨̢̡̣̩̣̥

The   end  result  must  be  that: ̡̨̢̡̣̩̣̥

  1. The build  user  can    type   “./gradlew run”  to  have  the  core application  run,  and    be   able  to  load  the  plugins/scripts. ̡̨̢̡̣̩̣̥

    ̡̨̢̡̣̩̣̥

  2. The build  user  can    also   type  “./gradlew build”,  and  unpack  and run  the  core    application’s   distribution  .zip  file,  to  achieve the  same  effect. ̡̨̢̡̣̩̣̥

    ̡̨̢̡̣̩̣̥

The    keymap   File

Keep  the  keymap  file  in the  core  application’s    base   directory  –  the  directory  containing the  core  application    subproject. ̡̨̢̡̣̩̣̥

You  can  insert    a   debugging  statement  to  print  the value  of  System.getProperty("user.dir"),    if   you  need  to  see  what the  “current  directory”    actually   is. ̡̨̢̡̣̩̣̥
